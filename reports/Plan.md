# Планирование тестирования приложения «Новости хосписа»

---

## 1. Определение границ приложения

### Что входит в тестируемый контур

- Клиентское мобильное приложение Android со следующим набором экранов:
  - главный экран со сводкой новостей;
  - лента новостей;
  - панель управления;
  - создание/рекактирование новости;
  - раздел с тематическими цитатами;
  - информационный экран "О приложении".
- Базовое взаимодействие с серверной частью: получение списка новостей, загрузка цитат, добавление новостей, удаление новостей, получение информации о приложении.

---

## 2. Реализованный функционал

### Модуль «Новости»

- Просмотр списка новостей:
  - отображение заголовка, описания;
  - бесконечная прокрутка или пагинация.
- Работа с новостью:
  - добавление новости;
  - удаление новости;
  - редактирование новости.

### Модуль «Тематические цитаты»

 - отображение цитат;
 - бесконечная прокрутка или пагинация.

### Модуль «О приложении»

- Статический экран с информацией:
  - название и версия приложения;
  - ссылка на "Политика конфиденциальности";
  - ссылка на "Пользовательсое соглашение".
- Переход на этот экран из меню.

### Общие проверки

- Навигация между разделами через бургер меню.
- Обработка поворота экрана и смены ориентации на всех экранах.
- Реакция на прерывания (входящий звонок, сворачивание приложения).
- Поведение при отсутствии интернет-соединения.

---

## 3. Стратегия тестирования и типы проверок


- **Функциональное тестирование** – основной массив, покрывает все пользовательские сценари.
- **Тестирование пользовательского интерфейса (UI/UX):**
  - соответствие макетам;
  - удобство использования.
- **Тестирование совместимости:**
  - различные версии ОС, разрешения экранов, диагонали;
  - различные производители оболочек (Samsung, Xiaomi и т.д.).
- **Локализация и доступность:** корректность текстов, контрастность.
- **Сетевое тестирование:** сброс соединения, переход между Wi-Fi/мобильными данными.

## 4. Интервальная оценка с учётом рисков

Оценка времени на тестирование и на внедрение автоматизации.

**Оценка ручного тестирования:**
* **Оптимистичная:** 2-3 рабочих дня.
* **Пессимистичная:** 4-5 рабочих дней.
* *Буфер на риски:* +20% к времени (закладывается 1 день).

**Оценка внедрения автоматизации (первичный цикл):**
* **Написание автотестов :** 2-3 недели.
* **Риски, влияющие на сроки:** 
  * *Нестабильность тестового окружения/API* .

## 5. Техники тест-дизайна с обоснованием

Для минимизации количества тест-кейсов при максимальном покрытии применяются следующие техники:

1. **Классы эквивалентности и Граничные значения:**
   * *Обоснование:* Идеально подходят для модуля «Создание/редактирование новости». Позволяют проверить лимиты символов в заголовке и описании (валидные/невалидные длины, пустые поля, спецсимволы).
2. **Тестирование по сценариям использования:**
   * *Обоснование:* Необходимо для проверки сквозных бизнес-процессов. Например: «Создание новости -> Публикация -> Отображение в ленте -> Редактирование -> Удаление».
3. **Переходы состояний:**
   * *Обоснование:* Критично для проверки сетевого взаимодействия и жизненного цикла приложения. Состояния: *Онлайн -> Офлайн -> Онлайн*, *Приложение в фоне -> Входящий звонок -> Возврат в приложение*.
4. **Исследовательское тестирование (Exploratory Testing):**
   * *Обоснование:* Необходимо для оценки UI/UX, удобства использования (usability) и поиска неочевидных багов, которые не покрываются формальными сценариями.

## 6. Перечень и описание возможных рисков при автоматизации

1. **Фрагментация Android-устройств:**
   * *Описание:* Автотесты, идеально работающие на эмуляторе или одном реальном устройстве, могут падать на устройствах с кастомными оболочками (MIUI, OneUI) из-за особенностей отрисовки UI или агрессивного энергосбережения.
2. **Хрупкость локаторов (Flaky Locators):**
   * *Описание:* Изменение верстки разработчиками приведет к падению UI-тестов.
3. **Изменение системных настроек устройства (Dark Mode, размер шрифта, язык):**
   * *Описание:* Пользователи могут использовать тёмную тему, увеличенный размер системного шрифта или нестандартный язык интерфейса. Это может ломать вёрстку, скрывать элементы или менять тексты, на которые завязаны локаторы.
4. **Проблемы с ожиданиями:**
   * *Описание:* Тестовый скрипт выполняется быстрее, чем мобильное приложение успевает отрисовать интерфейс, обработать данные или завершить фоновые процессы. Автотест может пытаться взаимодействовать с элементом, который еще не появился в дереве, еще не стал видимым или не перешел в состояние clickable. Это приводит к падениям с ошибками NoSuchElementException или к некорректным ассертам.

## 7. Перечень необходимых специалистов для автоматизации

Для полноценного внедрения и поддержки автоматизации потребуются:

1. **QA Automation Engineer (SDET):** Разработка фреймворка, написание автотестов, интеграция с CI/CD.
2. **QA Manual (Тестировщик):** Поддержка актуальности чек-листов, ручная проверка перед автоматизацией.
3. **CI/CD Engineer (Опционально, может быть совмещено с SDET):** Настройка пайплайнов (GitLab CI, Jenkins, GitHub Actions)

## 8. Сортировка тест-кейсов по степени автоматизации

Разделение проверок на те, что будут автоматизированы, и те, что останутся в ручном исполнении.

**Планируется автоматизировать (UI + API):**
* **Smoke-тесты:** Запуск приложения, авторизация (если есть), переход по всем пунктам бургер-меню, отображение элементов.
* **CRUD Новости:** Создание, редактирование, удаление новости.

**Останется в ручном исполнении:**
* **UI/UX и верстка:** Визуальное соответствие макетам, проверка шрифтов, отступов, адаптивности под разные диагонали (автоматизация визуальных проверок на разных устройствах слишком дорога).
* **Прерывания:** Входящий звонок, сворачивание, push-уведомления.
* **Доступность:** Проверка контрастности, озвучки скринридером.
* **Исследовательское тестирование:**
* **Локализация:** Проверка контекста переводов, отсутствие обрезанных текстов.

## 9. Тестовая документация на проекте


**Ручная тестовая документация:**
1. **План тестирования:** Текущий документ.
2. **Чек-листы и Тест-кейсы:** Включают шаги, ожидаемые результаты, тестовые данные.
3. **Баг-репорты:**  Содержат шаги, ожидаемый/фактический результат, логи, скриншоты/видео, информацию об устройстве.
4. **Отчет о тестировании:** Формируется по итогам релиза.

**Документация и артефакты автоматизации:**
1. **Кодовая база автотестов:** Включает сами тесты, страницы (Page Objects), утилиты для работы с API и БД, конфиги.
2. **Конфигурации CI/CD:** Скрипты пайплайнов для запуска тестов.
3. **Отчеты о прогоне автотестов:** Интеграция с **Allure Report**. Генерируется автоматически после каждого прогона. Содержит графики, историю падений, скриншоты на момент ошибки и логи.
